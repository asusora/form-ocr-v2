2. 前端测试配置方法。
3. 自动化测试流程。

- `Vue 3`
- `TypeScript`

## 3. 前端运行前提

在启动前端之前，需要确认以下前提已经满足：

### 3.1 后端服务已经启动

前端依赖后端提供 `/api` 接口。至少需要保证以下接口可访问：

```bash
GET /api/health
GET /api/templates
POST /api/templates
POST /api/recognitions
GET /api/recognitions/{id}
```

如果后端未启动，前端页面可以打开，但模板列表、模板创建、识别轮询等功能都会失败。

### 3.2 Node.js 环境可用

建议版本：

- Node.js `18+`
- npm `9+`

检查命令：

```bash
node -v
npm -v
```

### 3.3 前端依赖已安装

首次进入项目后，执行：

```bash
cd frontend
npm install
```

## 4. 前端环境变量配置说明

前端的接口访问行为由以下环境变量决定：

### 4.1 `VITE_API_BASE_URL`

作用：

- 控制浏览器真正发请求时使用的 API 前缀。

默认值：

```bash
/api
```

适用场景：

- 前后端同域部署。
- 本地开发时通过 Vite 代理转发到后端。

### 4.2 `VITE_API_PROXY_TARGET`

作用：

- 仅在本地 `vite dev` 开发服务器下生效。
- 控制 `/api` 代理到哪个后端地址。

默认值：

```bash
http://localhost:8020
```

### 4.3 推荐配置方式

复制示例文件：

```bash
cd frontend
copy .env.example .env.local
```

示例内容如下：

```bash
VITE_API_BASE_URL=/api
VITE_API_PROXY_TARGET=http://localhost:8020
```

## 5. 不同部署场景下的配置方法

### 5.1 场景一：本地开发联调

适用条件：

- 前端运行在 `http://127.0.0.1:5174`
- 后端运行在 `http://localhost:8020`

建议配置：

```bash
VITE_API_BASE_URL=/api
VITE_API_PROXY_TARGET=http://localhost:8020
```

说明：

- 浏览器访问前端地址。
- 前端把 `/api` 请求交给 Vite 代理，再转发到后端。
- 这是默认推荐方式。

### 5.2 场景二：前后端同源部署

适用条件：

- Nginx、网关或后端服务已经把前端页面和 `/api` 同时暴露在同一个域名下。

建议配置：

```bash
VITE_API_BASE_URL=/api
```

说明：

- 生产构建后不需要再使用 Vite 代理。
- 浏览器直接向当前域名下的 `/api` 发请求。

### 5.3 场景三：前后端跨域部署

适用条件：

- 前端和后端不在同一个域名或端口下。

建议配置：

```bash
VITE_API_BASE_URL=http://your-backend-host:8020/api
```

说明：

- 此时不能依赖 `/api` 相对路径。
- 需要把后端完整地址直接写入 `VITE_API_BASE_URL`。
- 同时后端必须正确配置 CORS。

## 7. 前端页面与业务流程说明

当前前端包含三个主页面。

### 7.1 模板列表页

路由：

```bash
/
```

页面作用：

- 查看模板列表。
- 新建模板。
- 进入模板编辑页。
- 基于模板创建识别任务。

对应文件：

- `frontend/src/views/TemplateListView.vue`

主要操作：

1. 点击“新建模板”。
2. 上传空白模板 PDF。
3. 创建成功后自动跳转到模板编辑器。
4. 在模板行点击“编辑模板”进入模板编辑。
5. 在模板行点击“去识别”上传待识别 PDF，创建识别任务。

### 7.2 模板编辑页

路由：

```bash
/templates/:id
```

页面作用：

- 浏览模板页图。
- 在页面上画框新增字段。
- 调整字段坐标。
- 配置字段名、字段类型、选项、表格列等信息。
- 保存模板字段。
- 查看自动生成的锚点预览。

对应文件：

- `frontend/src/views/TemplateEditorView.vue`

支持的字段类型：

- `text`
- `multiline_text`
- `date`
- `checkbox`
- `option_select`
- `signature`
- `table`

模板编辑页的交互规则：

1. 先点击“画框新增字段”。
2. 在左侧 PDF 页图上拖拽鼠标画框。
3. 选中字段后，在右侧“字段配置”中填写字段名、显示名和类型。
4. 如果是 `option_select`，需要配置选项值与匹配标签。
5. 如果是 `table`，需要配置列定义和行检测方式。
6. 点击“保存模板字段”后，后端会保存字段并自动生成锚点。

### 7.3 识别校对页

路由：

```bash
/recognitions/:id
```

页面作用：

- 显示识别任务状态。
- 自动轮询待处理任务。
- 查看每个字段的识别结果。
- 拖拽调整字段框。
- 手工修改识别结果。
- 对单字段执行“重新识别”。
- 保存校对结果。
- 导出 JSON 和 Excel。

对应文件：

- `frontend/src/views/RecognitionView.vue`

识别页状态说明：

- `pending`：任务已创建，等待开始。
- `processing`：后端正在 OCR、对齐和抽取。
- `success`：识别成功，可编辑和导出。
- `failed`：识别失败，页面展示错误信息。

## 8. 详细使用流程

以下流程适合作为项目日常使用流程。

### 8.1 使用流程一：创建模板

1. 进入模板列表页 `/`。
2. 点击“新建模板”。
3. 输入模板名称。
4. 可选填写模板描述。
5. 选择空白模板 PDF。
6. 根据清晰度调整 `renderDpi`，默认推荐 `200`。
7. 点击“创建模板”。
8. 页面会跳转到模板编辑页。

期望结果：

- 模板创建成功。
- 模板详情已经从后端加载完成。
- 可以看到页数、页图和当前字段数。

### 8.2 使用流程二：配置模板字段

1. 在模板编辑页点击“画框新增字段”。
2. 在左侧 PDF 页图上拖拽生成字段框。
3. 右侧会选中新字段。
4. 配置字段名和显示名。
5. 选择字段类型。
6. 如有需要，手工微调 `x1/y1/x2/y2`。
7. 如是单选项字段，填写候选值和匹配标签。
8. 如是表格字段，补充列定义和行检测策略。
9. 重复以上步骤，直到整份模板字段配置完成。
10. 点击“保存模板字段”。

期望结果：

- 保存后字段不再是草稿状态。
- 当前字段可在锚点预览中看到自动生成的锚点。

### 8.3 使用流程三：创建识别任务

1. 返回模板列表页。
2. 在目标模板行点击“去识别”。
3. 上传待识别 PDF。
4. 点击“开始识别”。
5. 页面跳转到识别校对页。
6. 页面自动轮询任务状态。

期望结果：

- 识别页先显示 `pending` 或 `processing`。
- 后端完成任务后自动刷新为 `success`。

### 8.4 使用流程四：识别结果校对

1. 在识别页左侧查看当前页字段框。
2. 点击某个框或右侧字段卡片选中字段。
3. 若框位置不准，直接在左侧拖动框。
4. 若框大小不准，拖动右下角缩放手柄。
5. 在右侧修改字段值。
6. 若 OCR 结果明显错误，点击“重新识别”。
7. 全部确认后点击“保存结果”。

期望结果：

- 被修改的字段状态会变成 `manual_adjusted`。
- 保存后数据写回后端。

### 8.5 使用流程五：导出结果

1. 在识别任务状态为 `success` 时，点击“下载 JSON”。
2. 或点击“下载 Excel”。

期望结果：

- 浏览器打开或下载导出文件。
- JSON 用于系统对接。
- Excel 用于人工复核或业务交付。

## 9. 前端测试配置文档

### 9.1 当前测试框架

当前前端采用：

- `Vitest`：测试运行器。
- `@vue/test-utils`：Vue 组件挂载与交互工具。
- `jsdom`：浏览器 DOM 模拟环境。

配置位置：

- `frontend/vite.config.ts`

当前配置要点：

- `environment: 'jsdom'`
- `setupFiles: ['./tests/setup.ts']`
- `css: true`
- `globals: true`

### 9.2 测试初始化文件

文件：

```bash
frontend/tests/setup.ts
```

作用：

1. 注册 `ElementPlus` 插件。
2. 对 `RouterLink`、`RouterView`、`Transition`、`Teleport` 做测试桩处理。
3. 为 `ResizeObserver` 提供 mock，避免在 `jsdom` 中报错。

这意味着：

- 当前测试适合组件级和 store 级测试。
- 不依赖真实浏览器。
- 不依赖真实后端服务。

### 9.3 当前已有自动化测试覆盖

当前测试文件如下：

| 文件 | 覆盖内容 |
| --- | --- |
| `frontend/tests/components/FieldConfigPanel.spec.ts` | 字段配置面板在 `option_select` 类型下是否展示选项配置区 |
| `frontend/tests/components/RecognitionFieldPanel.spec.ts` | 识别字段面板是否正确渲染文本字段信息 |
| `frontend/tests/unit/pdf-canvas.spec.ts` | 画框相关的坐标缩放、归一化、边界裁剪逻辑 |
| `frontend/tests/unit/recognition.store.spec.ts` | 识别 store 是否能加载识别详情 |
| `frontend/tests/unit/template-list.store.spec.ts` | 模板列表 store 是否能拉取模板摘要 |

### 9.4 当前测试能力边界

当前仓库已经具备：

- 单元测试。
- 组件渲染测试。
- Store 行为测试。

当前仓库尚未具备：

- 端到端测试。
- 真实浏览器自动化测试。
- 基于真实后端联调的前端自动验收测试。

因此当前推荐的测试策略是：

1. 自动化测试负责兜住基础逻辑和组件渲染。
2. 手工测试负责验证完整业务流程。

## 11. 推荐测试流程

### 11.1 开发自测流程

适用于前端开发改动后本地验证。

步骤如下：

1. 执行 `npm run test`，确保自动化测试通过。
2. 执行 `npm run build`，确保类型检查和构建通过。
3. 执行 `npm run dev`，打开本地页面。
4. 根据改动影响范围进行手工验证。

推荐判断标准：

- 自动化测试通过。
- 构建通过。
- 受影响页面功能可正常操作。

适用于前后端一起验证接口联动。

步骤如下：

1. 确认后端服务已启动。
2. 在 `frontend/.env.local` 中配置正确的 API 地址。
3. 启动前端：`npm run dev`。
4. 打开模板列表页。
5. 按“模板创建 -> 模板编辑 -> 创建识别 -> 识别校对 -> 导出结果”的完整链路执行。

### 11.3 交付前回归测试流程

适用于版本发布前统一验收。

建议顺序如下：

1. 自动化测试回归：


2. 构建回归：

```bash

npm run build
```

3. 手工业务回归：

- 新建模板

- 保存模板字段

## 13. 常见问题与排查方法

### 13.1 页面打开了，但列表接口报错

排查项：

1. 后端是否启动。
2. `VITE_API_BASE_URL` 是否正确。
3. 本地开发时 `VITE_API_PROXY_TARGET` 是否正确。
4. 浏览器网络面板里 `/api/templates` 是否请求到了错误地址。

### 13.2 开发环境下跨域报错

原因：

- 直接把 `VITE_API_BASE_URL` 配成了跨域地址，但后端没有放开 CORS。

解决方式：

- 本地联调优先使用 `/api` + `VITE_API_PROXY_TARGET` 代理模式。

### 13.3 测试运行时报 `ResizeObserver is not defined`

原因：

- 测试环境没有加载 `tests/setup.ts`。

解决方式：

- 确认测试由 `vitest` 运行。
- 确认 `vite.config.ts` 中 `setupFiles` 指向 `./tests/setup.ts`。

### 13.4 自动化测试通过，但业务流程仍失败

原因：

- 当前自动化测试主要覆盖组件和 store，不覆盖真实联调链路。

解决方式：

- 执行完整手工测试流程。
- 特别验证模板创建、识别轮询和导出接口。

### 13.5 识别页一直停留在处理中

排查项：

1. 后端识别任务是否真正启动。
2. 后端 `/api/recognitions/{id}` 是否一直返回 `processing`。
3. OCR 运行环境是否正常。
4. 后端日志是否有异常。
